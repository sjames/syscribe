### 7.1 The TARA workflow

Threat Analysis and Risk Assessment follows a chain analogous to HARA, but for security:

```
Asset (ASSET-*)                ← what is being protected (§15.3)
    ↓ assets:
DamageScenario (DS-*)          ← what damage occurs if the asset is compromised (§15.4)
    ↓ damageScenarios:
ThreatScenario (TS-*)          ← how an attacker achieves the damage (§15.5–15.7)
    ↓ threatScenarios:
CybersecurityGoal (CSG-*)      ← what the system must do to prevent the threat (§15.8)
    ↓ implementsGoals:
SecurityControl (SC-*)         ← the technical mechanism implementing the goal (§15.9)
    ↓ derivedFromSecurityGoal:
Requirement (REQ-*)            ← technical requirement derived from the CSG
    ↓ verifies:
TestCase (TC-*)                ← evidence of goal satisfaction
```

!!! note "Two authoring styles for TARA elements"
    This guide shows TARA elements as **standalone files** — one `.md` per `Asset`,
    `DamageScenario`, `ThreatScenario`, etc. This is the recommended approach for large or
    evolving models because each element is independently diffable and reviewable.

    An alternative is the **`TARASheet` container** (`type: TARASheet`): a single file that
    holds all four TARA tiers as inline tables. The parser expands each row into a first-class
    element at load time, so all validation rules and cross-reference checks apply identically.
    The `TARASheet` approach suits small models or initial drafts where all TARA artefacts fit
    comfortably in one file. See the [Safety & Security Analysis guide](../model-guide/safety-analysis.md)
    for the `TARASheet` format and `syscribe template TARASheet` for a ready-to-fill skeleton.

### 7.2 Assets (ISO/SAE 21434 §15.3)

Assets are what an attacker wants to compromise. Define them explicitly before defining
damage scenarios.

```yaml
---
type: Asset
id: ASSET-KERNEL-001
name: "Thread Control Block (TCB)"
status: approved
cybersecurityProperties:
  - integrity
  - availability
assetOwner: SabatonRt::Software::KernelCore
relatedSafetyGoal: Safety::SG-KERNEL-002-SpatialIsolation
---

The TCB holds the thread's register save area, stack pointer, priority, and
IPC waitlist link. Corruption enables priority inversion or control-flow hijack.
```

`cybersecurityProperties` must be one or more of: `confidentiality`, `integrity`,
`availability`, `authenticity` (E863 otherwise).

W810 fires if an Asset is not referenced by any `DamageScenario.assets:` — that means the
asset has been identified but no attacker has been imagined damaging it yet.

### 7.3 Damage scenarios (§15.4)

```yaml
---
type: DamageScenario
id: DS-KERNEL-001
name: "Attacker corrupts scheduler state causing priority inversion"
status: approved
damageSeverity: severe
impactCategories:
  - safety
  - operational
assets:
  - Security::Assets::ASSET-KERNEL-001
  - Security::Assets::ASSET-KERNEL-002
hazardRef: Safety::SG-KERNEL-001-SchedulingIntegrity   # safety↔security co-link
---
```

`hazardRef` creates the cross-domain link. W030 fires if a safety-tagged damage scenario
lacks a `hazardRef`. View the full safety↔security overlap:

```bash
syscribe -m model co-analysis
```

### 7.4 Threat scenarios and risk (§15.5–15.8)

```yaml
---
type: ThreatScenario
id: TS-KERNEL-001
name: "Local attacker overwrites TCB via stack overflow"
status: approved
attackFeasibility: high
attackVector: local
damageScenarios:
  - Security::DamageScenarios::DS-KERNEL-001
riskTreatment: reduce
residualRisk: "Low after PSPLIM stack-limit enforcement (REQ-STKLIM-001)"
---
```

Risk is derived automatically: `score = damageSeverity + attackFeasibility` → low / medium /
high / critical. W031 fires if a high/critical-risk threat has no `riskTreatment` and is not
addressed by a `CybersecurityGoal`. Gate this in CI:

```bash
syscribe -m model validate --deny W031
```

Run the full risk matrix:

```bash
syscribe -m model cyber-risk
```

### 7.5 Attack trees (§15.7)

Attack trees decompose a threat scenario into the attack steps an adversary must execute.
Feasibility rolls up automatically — AND gates take MIN (sequential steps are hard as the
hardest step), OR gates take MAX (attacker picks easiest path). This mirrors FTA logic but
for adversarial feasibility rather than random hardware failure.

**Directory layout** (W036 if violated):

```
model/Security/AttackTrees/
  AT-KERNEL-001.md             ← AttackTree
  AT-KERNEL-001/
    _index.md
    ATG-KERNEL-001.md          ← AttackTreeGate (root gate)
    ATS-KERNEL-001.md          ← AttackStep (leaf)
    ATS-KERNEL-002.md          ← AttackStep (leaf)
```

```yaml
---
type: AttackTree
id: AT-KERNEL-001
name: "TCB corruption via stack overflow"
threatRef: Security::ThreatScenarios::TS-KERNEL-001
gate: OR           # root children joined by this gate
children:
  - Security::AttackTrees::AT-KERNEL-001::ATG-KERNEL-001
---
```

```yaml
---
type: AttackTreeGate
id: ATG-KERNEL-001
gateKind: AND     # attacker must execute all children in sequence
children:
  - Security::AttackTrees::AT-KERNEL-001::ATS-KERNEL-001
  - Security::AttackTrees::AT-KERNEL-001::ATS-KERNEL-002
---
```

```yaml
---
type: AttackStep
id: ATS-KERNEL-001
name: "Induce deep recursion in a low-privilege thread"
attackFeasibility: high
---
```

W035 fires if the computed tree feasibility disagrees with the `ThreatScenario.attackFeasibility`.
This is the security equivalent of W033 — it catches inconsistencies between your qualitative
threat model and your tree decomposition.

### 7.6 Cybersecurity goals and CAL level (§15.8)

```yaml
---
type: CybersecurityGoal
id: CSG-KERNEL-001
name: "Prevent scheduler starvation via priority manipulation"
status: approved
securityProperty: availability
calLevel: CAL4
threatScenarios:
  - Security::ThreatScenarios::TS-KERNEL-001
---
```

W032 fires if `calLevel` is below the minimum implied by the threat risk level
(low→CAL1, medium→CAL2, high→CAL3, critical→CAL4). W802 fires if no `SecurityControl`
implements the goal; W804 fires if no requirement is derived from it.

### 7.7 Security controls and requirements

```yaml
---
type: SecurityControl
id: SC-KERNEL-001
name: "PSPLIM stack-limit register enforcement"
status: approved
controlType: prevention
implementsGoals:
  - Security::Goals::CSG-KERNEL-001
---
```

Requirements derived from the cybersecurity goal carry the link:

```yaml
derivedFromSecurityGoal: Security::Goals::CSG-KERNEL-001-SchedulingAvailability
```

### 7.8 Security verification (§13.3)

Group all security test cases under a security TestPlan with `scope: security`. The plan
`demonstrates:` the CSGs it provides evidence for. Individual test cases carry
`securityTestMethod:` from the 21434 §13.3 vocabulary.

```yaml
---
type: TestPlan
id: TP-SEC-001
name: "Kernel Security Verification Plan"
status: approved
scope: security
demonstrates:
  - Security::Goals::CSG-KERNEL-001
  - Security::Goals::CSG-KERNEL-002
selection:
  tags: [security]
---
```

```yaml
# On a relevant TestCase:
tags: [security]
securityTestMethod: penetration_test
```

W612 fires if the plan's effective test case set is empty (tags don't match any TestCase).

### 7.9 Vulnerability tracking

Track discovered CVEs with `VulnerabilityReport`:

```yaml
---
type: VulnerabilityReport
id: VR-KERNEL-001
name: "Stack overflow via unbounded recursion in thread entry function"
status: open
cvssScore: 7.2
mitigatedBy:
  - Security::Controls::SC-KERNEL-001
affectedElements:
  - SabatonRt::Software::KernelCore
---
```

W803 fires when status is `open` — use this to track unresolved vulnerabilities in the gate.

### 7.10 What syscribe does NOT compute for security

Unlike the safety side (where `syscribe metrics` gives quantitative PMHF / SPFM / LFM),
the security computation remains qualitative — this mirrors ISO/SAE 21434 itself, which
defines CAL 1–4 (qualitative risk levels) rather than a numeric probability target.

The attack tree feasibility rollup and the `cyber-risk` risk matrix are the security
equivalents of the fault tree probability rollup, but they produce `low / medium / high /
critical` verdicts rather than numeric failure rates.
