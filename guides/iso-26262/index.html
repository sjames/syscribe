    Part V — Safety: ISO 26262 (Automotive)¶

    GUIDE · PART V — ISO 26262

    Disclaimer

    Syscribe is a modeling and documentation tool provided "as-is" without warranty of any kind. The authors and contributors accept no responsibility or liability for the use of this tool in safety-critical, life-critical, or mission-critical applications.

    Output from Syscribe — including validation results, generated templates, traceability reports, and analysis artifacts — must be independently reviewed and verified by qualified engineers before use in any certification, regulatory submission, or safety case. Compliance with standards such as ISO 26262, IEC 61508, ISO 13849-1, ISO/SAE 21434, IEC 61025, DO-178C, or any other functional safety or cybersecurity standard remains the sole responsibility of the user and their organisation.

    This tool does not replace a certified safety process, a qualified safety engineer, or a formal assessment body.

    5.1 The complete workflow¶

    HazardousEvent (HE-*)          ← from HARA — operational situation + hazard
    ↓ hazardousEvents:
SafetyGoal (SG-*)              ← top-level safety requirement, ASIL derived from S/E/C
    ↓ derivedFromSafetyGoal:
Requirement (REQ-*)            ← technical requirements, ASIL inherited/decomposed
    ↓ satisfies: (on PartDef)
PartDef (architecture)         ← design element implementing the requirement
    ↓ verifies:
TestCase (TC-*)                ← evidence that the requirement is met
    ↓
FaultTree / FaultTreeEvent     ← quantitative FMEDA (SPFM, LFM, PMHF)
Argument (ARG-*)               ← GSN safety case linking everything together
AssumptionOfUse (AOU-*)        ← safety manual / SRACs
ConfirmationMeasure (CM-*)     ← review records, independence assessment

    5.2 HARA — HazardousEvent¶

    The Hazard Analysis and Risk Assessment begins with HazardousEvent elements. ASIL is derived by syscribe from the S/E/C parameters — you do not set it manually.

    ---
type: HazardousEvent
id: HE-KERNEL-001
name: "Scheduler starvation causes safety monitor thread to miss deadline"
status: approved
severity: S3
exposure: E4
controllability: C2
---

**Hazard**: Priority inversion or starvation in the kernel scheduler.  
**Operational situation**: Vehicle in motion at highway speed; safety monitor
thread rate-monotonic at 10 ms period.  
**Derived ASIL**: D (S3 × E4 × C2 → ASIL D per ISO 26262-3 Table 4).

    The validator derives ASIL D and propagates it as the required integrity level for the associated SafetyGoal. show HE-KERNEL-001 displays the derivation.

    5.3 SafetyGoal¶

    SafetyGoals are the top-level safety requirements derived from the HARA. Each SafetyGoal carries the normative integrity level and the ftti: (fault-tolerant time interval — ISO 26262 §8.4.2.3).

    ---
type: SafetyGoal
id: SG-KERNEL-001
name: "Kernel scheduler shall not starve any safety-relevant thread"
status: approved
asilLevel: D
safeState: "Hardware watchdog reset; vehicle enters safe stop state"
ftti: "Bounded by the watchdog supervision period — typically ≤ 10 ms at 100 Hz tick"
hazardousEvents:
  - Safety::HE-KERNEL-001
---

    W806 fires if hazardousEvents: is absent. W801 fires if no integrity level is set.

    5.4 Derived requirements¶

    Requirements derived from a safety goal carry the same ASIL level. Use derivedFromSafetyGoal: to express this link directly (in addition to derivedFrom: for hierarchical decomposition):

    ---
type: Requirement
id: REQ-SCHED-001
name: "Scheduler shall select highest-priority ready thread at every scheduling point"
status: approved
asilLevel: D
derivedFrom:
  - Safety::SG-KERNEL-001-SchedulingIntegrity
derivedFromSafetyGoal: Safety::SG-KERNEL-001-SchedulingIntegrity
breakdownAdr: Decisions::ADR-SCHED-001
verificationMethod: test
---

    5.5 Fault Tree Analysis (FTA)¶

    FTA decomposes a top-level failure (the safety goal violation) into hardware and software contributing causes. syscribe implements a three-level structure:

    FaultTree (FT-*)               ← the tree root, linked to a SafetyGoal
  FaultTreeGate (FTG-*)        ← AND/OR intermediate events
    FaultTreeEvent (FTE-*)     ← basic events (leaf failures)

    FaultTree root:

    ---
type: FaultTree
id: FT-KERNEL-001
name: "Scheduler fails to execute highest-priority thread"
status: approved
topEvent: "Priority-ordering failure leading to SG-KERNEL-001 violation"
safetyGoal: Safety::SG-KERNEL-001-SchedulingIntegrity
gate: OR          # top event occurs if any child occurs
children:
  - FTA::FT-KERNEL-001::FTG-KERNEL-001
  - FTA::FT-KERNEL-001::FTG-KERNEL-002
---

    FaultTreeGate:

    ---
type: FaultTreeGate
id: FTG-KERNEL-001
name: "Ready queue corruption"
gateKind: OR
children:
  - FTA::FT-KERNEL-001::FTE-KERNEL-101
  - FTA::FT-KERNEL-001::FTE-KERNEL-102
---

    FaultTreeEvent (basic event with FMEDA fields):

    ---
type: FaultTreeEvent
id: FTE-KERNEL-101
name: "Stack overflow corrupts TCB ready-queue pointer"
eventKind: basic
failureRate: 5.0e-9          # failures per hour (IEC 61508-7 T.C.6 estimate)
diagnosticCoverage: 0.97     # fraction of faults detected by the safety mechanism
latentDiagnosticCoverage: 0.75  # fraction detected by the latent fault mechanism
---

Stack overflow undetected by PSPLIM corrupts the TCB's `next_ready` pointer.

**Failure rate**: 5e-9/h — PSPLIM is a hardware mechanism on ARMv8-M; this is
a first-order software estimate for the residual probability of PSPLIM
misconfiguration or stack frame miscalculation.

**DC = 0.97**: TC-STKLIM-FAULT-HIL-001 validates the full overflow-to-trap-to-
reaction path on hardware. TC-PORT-STKLIM-001 covers register programming.

**DCl = 0.75**: Watchdog deadman backstop provides latent coverage. 25% residual
represents the window between a silent TCB corruption and the next scheduler
selection that would expose it.

    Nesting rule (W900): gates and events must live in a subdirectory named after the FaultTree file:

    model/Safety/FTA/
  FT-KERNEL-001.md
  FT-KERNEL-001/
    _index.md
    FTG-KERNEL-001.md
    FTE-KERNEL-101.md

    Render the tree as Mermaid:

    syscribe -m model fault-tree render FT-KERNEL-001

    5.6 Quantitative FMEDA metrics¶

    Once failureRate, diagnosticCoverage, and latentDiagnosticCoverage are populated on the basic events, syscribe computes SPFM, LFM, and PMHF per safety goal:

    syscribe -m model metrics

    Example output:

    | SafetyGoal     | FaultTree     | SPFM   | LFM    | PMHF          | Verdict |
|---|---|---|---|---|---|
| SG-KERNEL-002  | FT-KERNEL-002 | 0.788  | 0.632  | 4.518e-9/h    | PASS    |
| SG-KERNEL-003  | FT-KERNEL-001 | 0.953  | 0.738  | 2.077e-9/h    | PASS    |

    For ISO 26262 ASIL D: targets are SPFM ≥ 0.99, LFM ≥ 0.90, PMHF < 1e-8/h. Gate PMHF failures in CI:

    syscribe -m model validate --deny W033

    Interpreting a sub-target SPFM: if SPFM is below target (as in SG-KERNEL-002 above), identify the FaultTreeEvent with the lowest diagnosticCoverage — that is the dominant gap. W033 details which event drives the miss. Use this to prioritise safety mechanism work.

    5.7 FMEA¶

    The FMEASheet element holds a Failure Mode and Effects Analysis. Each entry records a failure mode, its effect, severity, occurrence, detectability, and RPN.

    ---
type: FMEASheet
id: FMEA-KERNEL-001
name: "Kernel FMEA — scheduling and memory protection"
status: approved
safetyGoals: [SG-KERNEL-001, SG-KERNEL-002]
entries:
  - id: FM-KERNEL-001
    component: KernelScheduler
    failureMode: "Ready queue pointer corruption"
    localEffect: "Wrong thread selected"
    systemEffect: "Priority inversion; safety monitor starved"
    severity: 10
    occurrence: 2
    detectability: 1
    controls: [SC-KERNEL-001]
    faultTreeRef: FTE-KERNEL-101
---

    Render the RPN-ranked report:

    syscribe -m model fmea report FMEA-KERNEL-001

    5.8 Safety case (GSN)¶

    The Argument type implements GSN (Goal Structuring Notation) nodes. Each node is a claim, strategy, or solution that, together, constitute the safety case for a safety goal.

    ---
type: Argument
id: ARG-KERNEL-001
name: "Top claim: kernel scheduling is safe for ASIL D"
argKind: goal
status: approved
supports: Safety::SG-KERNEL-001-SchedulingIntegrity
supportedBy:
  - Safety::Arguments::ARG-KERNEL-002   # strategy
evidence:
  - Requirements::Safety::SchedulingIntegrity   # requirement package
  - Verification::L1_Host::SchedulerTests       # test cases
---

The kernel scheduler implements priority-preemptive scheduling that satisfies
SG-KERNEL-001 for all defined configurations.

    Visualise the full safety case tree for a goal:

    syscribe -m model safety-case SG-KERNEL-001

    5.9 Assumptions of Use (SRACs)¶

    AssumptionOfUse elements record the safety manual conditions that the integrator must fulfil. They appear in the safety case and are checked by co-analysis.

    ---
type: AssumptionOfUse
id: AOU-KERNEL-001
name: "Integrator shall not use priority 0 for safety-critical threads"
status: approved
---

Priority 0 is reserved for the idle thread. Safety-relevant threads **shall**
use priorities 1–7 to guarantee preemption by the safety monitor.

    5.10 Confirmation measures¶

    ConfirmationMeasure elements record formal reviews, audits, and independence assessments per ISO 26262-2 §6. They carry the independence level (I1, I2, I3).

    ---
type: ConfirmationMeasure
id: CM-KERNEL-001
name: "SIL4 Point-in-Time Safety Audit"
status: complete
cmKind: safety_assessment
independenceLevel: I2
completedDate: 2026-06-08
---

    ISO 26262-26 ASIL D formally requires I3 (different organisation). If your project is at I2, document it here and include it as a known gap in your safety plan.

    5.11 Key commands for ISO 26262¶

    syscribe -m model metrics                      # SPFM / LFM / PMHF per SafetyGoal
syscribe -m model safety-case                  # full GSN tree for all goals
syscribe -m model safety-case SG-KERNEL-001    # GSN tree for one goal
syscribe -m model fmea report FMEA-KERNEL-001  # RPN-ranked FMEA table
syscribe -m model fault-tree render FT-KERNEL-001  # Mermaid FTA diagram
syscribe -m model co-analysis                  # safety ↔ security overlap
syscribe -m model audit                        # readiness dashboard
syscribe -m model verification-depth --sil D --min-levels 2  # depth gate
